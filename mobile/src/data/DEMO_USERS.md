## Purpose of Demo Mode

The offline demo mode serves multiple purposes:

1. **Rapid Evaluation** - Allows stakeholders to immediately explore features without backend setup
2. **Development Testing** - Enables frontend developers to test UI/UX without backend dependencies
3. **Offline Functionality** - Demonstrates the app's offline capabilities and local data handling
4. **Training & Demos** - Provides a consistent, reliable environment for training sessions and product demonstrations
5. **Quality Assurance** - Facilitates UI testing and screenshot generation without network variability
6. **Performance Benchmarking** - Allows testing of app performance with realistic data loads

## Important Notes

### Demo Mode Limitations

1. **Read-Only Data** - Changes made during demo sessions are not persisted
2. **No Real-Time Updates** - Data is static and pre-generated
3. **Limited Date Range** - Sample data covers approximately 60 days
4. **No Network Calls** - All API calls are intercepted and return local data
5. **Fixed Relationships** - Parent-child relationships are pre-defined
6. **Static Tokens** - Authentication tokens are dummy values

### Backend Independence

The demo mode operates **completely offline** without requiring:
- Backend server running
- Database connection
- API endpoints
- Network connectivity
- External services
- Authentication server
- Cloud storage

### Data Structure

All demo data is defined in `/mobile/src/data/dummyData.ts` and includes:
- Complete TypeScript type definitions
- Realistic sample values
- Proper relationships between entities
- Date calculations for dynamic values
- Consistent formatting and conventions

### Customization

Developers can modify demo data by editing `dummyData.ts`:
- Adjust student/parent information
- Add more assignments or exams
- Modify attendance patterns
- Change gamification values
- Add additional children for parent account
- Update grades and performance metrics

### Security Considerations

- Demo credentials are for **development and testing only**
- Never use demo credentials in production
- Demo mode should be disabled in production builds
- Ensure proper authentication flow is implemented for production
- Demo data should not contain any real user information

## Troubleshooting

### Login Issues
- Ensure exact email and password match (case-sensitive)
- Check that demo mode is enabled in app configuration
- Verify authentication service is using dummy data provider

### Missing Data
- Check that `dummyData.ts` is properly imported
- Verify data structure matches TypeScript interfaces
- Ensure all required fields are populated

### Navigation Errors
- Confirm role-based routing is configured correctly
- Check that all screens are registered in navigation
- Verify permissions for each role

### Performance Issues
- Monitor for memory leaks with large datasets
- Check that list rendering is optimized (virtualization)
- Verify images are properly cached or mocked
